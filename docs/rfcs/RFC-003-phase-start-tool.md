# RFC-003: Phase Start Tool — Seamless Transition Between Phases

**Status:** Accepted
**Date:** 2026-03-11
**Author:** Niklas Karlsson + Claude (dialogically developed)

---

## Problem

### Phases 2a and 2b have "start" tools. Phases 3–6 do not.

When a researcher begins Phase 2a, they call `code_start`. The tool:
1. Loads methodology automatically
2. Finds the right transcript
3. Returns the first chunk of data
4. Logs a session start event

When they begin Phase 2b, `review_start` does the same: loads methodology, finds the first unreviewed segment, logs a session start.

**From Phase 3 onwards, this breaks down.** The researcher says "nästa steg" (next step) or "Fas 5" and Claude Desktop must:
1. Know that `methodology_load` exists (not obvious)
2. Know the correct phase string ("phase5", not "5" or "Phase 5")
3. Know which files are relevant (code extraction? theme definitions? which version?)
4. Know to log a session start

In practice, Claude Desktop often skips methodology loading entirely, jumps straight to the data, or loads the wrong phase. The researcher loses the guided experience that Phases 2a/2b provide.

### The gap in researcher experience

| Phase | How it starts | Methodology loaded? | Relevant files identified? |
|-------|--------------|--------------------|-----------------------------|
| 2a | `code_start` | Yes (automatic) | Yes (transcript + first chunk) |
| 2b | `review_start` | Yes (automatic) | Yes (transcript + first segment) |
| 3 | `phase3_extract_codes` | No | Partial (generates output files) |
| 4 | Researcher must know to call `methodology_load` | Only if researcher asks | No |
| 5 | Researcher must know to call `methodology_load` | Only if researcher asks | No |
| 6 | Researcher must know to call `methodology_load` | Only if researcher asks | No |

### What this is NOT

This is not about building methodology into tools. The tool does not tell the researcher what to do, does not suggest themes, does not impose a workflow. It provides two things:
1. The methodology document (to read, not to summarise)
2. A list of tool-generated files and transcripts in the project

---

## Design Principles

1. **Infrastructure, not method** — the tool loads documents and lists files. It does not guide analysis.
2. **One tool for all phases** — `phase_start` works for phases 3–6. For 2a/2b it redirects to the dedicated tools.
3. **No duplication with existing tools** — if `code_start` or `review_start` already handles the phase, `phase_start` simply redirects.
4. **The tool knows what it generated** — tool outputs have known paths. The tool checks if they exist and lists them. No glob heuristics, no guessing.

---

## Proposed Design

### New tool: `phase_start`

```
Input:
  project_path: string   — Path to project directory (contains rta_config.yaml)
  phase: string          — "3", "4", "5", or "6" (also accepts "2a", "2b")

Output:
  methodology            — Full methodology document content (same as methodology_load)
  phase_scaffolding      — Work steps, reflection questions, AI countermeasures (phases 3–6)
  tool_outputs           — Tool-generated files that exist in the project
  transcripts            — Transcript list from rta_config.yaml with phase status
  instructions           — "SHOW methodology to researcher. Do NOT summarise."
```

### Tool outputs: the tool knows its own products

The tool checks for files at **known paths** — files generated by other tools in the system. No glob patterns, no heuristics.

| Tool output | Generated by | Path pattern |
|-------------|-------------|--------------|
| `phase3_code_extraction.md` | `phase3_extract_codes` | `{project}/phase3_code_extraction.md` |
| `phase3_code_summary.md` | `phase3_extract_codes` | `{project}/phase3_code_summary.md` |
| `*_coding_log.md` | `code_write_segment`, `review_write_note`, `review_revise_codes` | Same directory as transcript |
| `*_process_log.jsonl` | Various tools | Same directory as transcript |

Each file entry includes: path, size, last modified. Nothing more.

If a file does not exist, it is not listed. No "expected but missing" warnings — the researcher may not have reached that step yet.

### Transcripts from rta_config.yaml

Already tracked. Each transcript has `current_phase` status. The tool returns them as-is:

```
transcripts:
  - name: Skyttegatan_2_kb.md
    path: /path/to/transcript.md
    current_phase: phase2b_complete
```

### Redirect for Phase 2a/2b

If the researcher calls `phase_start(phase="2a")`, the tool returns:

```json
{
  "redirect": "phase2a_code_start",
  "message": "Phase 2a has a dedicated start tool. Call phase2a_code_start(file_path=...) instead."
}
```

Same for 2b → `phase2b_review_start`.

### Phase 3 special case

Phase 3 has `phase3_extract_codes` but no methodology auto-loading. `phase_start(phase="3")`:
1. Loads Phase 3 methodology
2. Lists transcripts (so the researcher sees which are coded)
3. Returns instruction: "When ready, call `phase3_extract_codes` to generate the code summary."

This fills the gap where Phase 3 currently skips methodology.

### Session logging

The tool logs a `session_start` event to the process log (best-effort), just like `code_start` and `review_start` do.

### Phase scaffolding

`phase_start` returns the same `phase_scaffolding` that `methodology_load` currently provides for phases 3–6 (work_steps, reflection_questions, ai_countermeasures). This is passed through from the existing `getPhaseScaffolding()` function — no duplication.

---

## What the tool does NOT do

- Does not suggest which files to read first
- Does not summarise file contents
- Does not propose themes, codes, or analytical structures
- Does not filter or select data
- Does not tell the researcher what a phase means — the methodology document does that
- Does not track researcher-created files — `list_files` exists for that

---

## Alternatives Considered

### Alternative A: Extend `methodology_load` with `relevant_files`

Add a `project_path` parameter to `methodology_load`. When provided, also scan for relevant files.

**Rejected because:** `methodology_load` is a focused tool (load a document). Adding project scanning changes its purpose. Also, `methodology_load` supports progressive loading (document_index) — adding file scanning to every document load is wasteful. Better to keep it focused and have a separate entry point.

### Alternative B: No new tool — improve init-instructions.md

Add clearer instructions for Claude Desktop about Phase 4–6 workflow.

**Rejected because:** This is the current approach and it does not work reliably. Claude Desktop does not consistently follow init-instructions for Phase 4–6 transitions. A tool call is more reliable than an instruction.

### Alternative C: Separate tools per phase (`phase4_start`, `phase5_start`, `phase6_start`)

**Rejected because:** The logic is identical — only the methodology document differs. One tool with a phase parameter is simpler and easier to maintain.

### Alternative D: Glob patterns for researcher-created files

Scan project directory for `*themes*`, `*definition*`, `*rapport*` etc.

**Rejected because:** Heuristic — may miss files with unexpected names or match unrelated files. Researcher knows their own files. `list_files` exists if they need to find them. YAGNI.

### Alternative E: Track all phase outputs in `rta_config.yaml`

Register every file created by `write_file` under a `phase_outputs` section.

**Rejected because:** Couples `write_file` to phase logic. Every save would require "which phase is this for?" — that is method in tools. Tool-generated files have known paths; researcher-created files are the researcher's business.

## Implementation

### New files
- `src/tools/phase_start.ts` — the tool implementation

### Modified files
- `src/server.ts` — register the new tool
- `src/tools/init.ts` — add to tool list
- `docs/mcp-usage/init-instructions.md` — update Phase 3–6 workflow section

### Changes to init-instructions.md

The "Phase 4–6 Workflow" section currently says:

> **When the researcher is ready for Phase 4, 5, or 6:**
> 1. Load the phase methodology: `methodology_load(phase="phase4")`

This changes to:

> **When the researcher is ready for Phase 3, 4, 5, or 6:**
> 1. Call `phase_start(project_path=..., phase="4")`
> 2. SHOW full methodology content to researcher — do not summarise
> 3. Review the list of tool outputs and transcripts returned by the tool
> 4. Work through the phase in conversation
